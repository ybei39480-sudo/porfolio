0. 执行摘要

闪动校园Pro 是高校广泛使用的健康跑打卡 App。它的防作弊机制设计得过于严格，导致大量正常跑步的用户被误判为作弊——78% 的受访同学曾被 GPS 漂移误判，85% 认为人脸识别弹窗频率过高，62% 在弱网环境下被误判。用户在使用过程中的核心体验是"跑完不算完，还要跟系统斗智斗勇"。

本文档提出一套体验优化方案，核心理念是把防作弊从"拦截正常用户"变为"精准识别异常行为"。主要做三件事：

1. 人脸识别从随机触发改为异常触发——正常跑步全程不打扰
2. 弱网定位补偿——短时信号丢失自动补全轨迹，不再误判
3. 三色状态提示面板——让用户实时了解跑步状态，消除不确定性焦虑

目标指标

| 指标 | 当前（调研估算） | 目标 |
|------|-----------------|------|
| 正常跑步人脸识别触发次数 | 单次 3-5 次（随机弹窗） | 0 次（仅异常触发） |
| GPS 漂移误判率 | 78% 用户曾被误判 | 降低至 10% 以下 |
| 弱网误判率 | 62% 用户曾被误判 | 降低至 15% 以下 |

> 以上数据来源于 32 人深度访谈和 50+ 份线上问卷，非闪动校园官方后台数据。


1. 产品背景

1.1 现在的问题

闪动校园Pro 作为高校健康跑打卡工具，承担着防止学生作弊（代跑、骑车等）的职责。但其防作弊机制在设计上偏向"宁可错杀、不可放过"的策略，导致大量正常跑步用户被误伤。

根据校内调研（[user_research.md](user_research.md)），问题集中在三个方面：

- GPS 漂移误判作弊：操场等开阔场景下 GPS 信号受环境影响产生随机漂移，系统将漂移判定为轨迹异常。78% 的受访同学曾被误判，每次误判意味着本次跑步成绩无效。
- 人脸识别弹窗频率过高：系统采用随机弹窗策略，单次跑步平均弹出 3-5 次。弹窗时用户需停下操作手机，跑步节奏被完全打断。85% 的同学认为弹窗频率过高。
- 弱网环境下缺乏容错机制：傍晚操场高峰期网络拥堵，定位信号频繁中断，信号丢失后系统直接判定作弊。62% 的同学曾在弱网环境下被误判。

1.2 机会分析

市面上成熟的跑步 App（Keep、悦跑圈、乐动力）在防作弊策略上普遍采用"异常检测"而非"随机拦截"，并配有轨迹平滑算法、离线缓存、申诉机制等功能。闪动校园Pro 的问题不在于防作弊本身，而在于防作弊逻辑过度简化——用简单规则覆盖所有场景，缺少对正常用户场景的考量。

如果能够借鉴成熟运动 App 的做法，将防作弊策略从"全员拦截"调整为"精准识别"，可以在不降低风控效果的前提下大幅提升正常用户的跑步体验。


2. 目标用户

用户 A：规律跑步者
特征：每天傍晚去操场跑步打卡，配速稳定在 5-6 分钟/公里。
最痛的点：GPS 偶尔漂移就被判作弊，平均每周白跑 1-2 次。
核心诉求：别误判正常跑步的人，让每次跑步都能顺利打卡。


用户 B：被动跑步者
特征：为了体育成绩被迫跑步打卡，配速 7-8 分钟/公里，偶尔停下来休息。
最痛的点：跑得慢也弹人脸识别，停下来系鞋带也弹，跑一次被打断好几次。
核心诉求：减少不必要的弹窗，让跑步过程顺畅一点。


3. 核心流程对比

3.1 现在（As-Is）

```
开始跑步 → GPS 正常采集轨迹
              ↓
        随机时刻弹出人脸识别（单次 3-5 次）
              ↓ 用户停下操作手机
              ↓ 验证通过才能继续
              ↓
        GPS 信号弱/漂移
              ↓
        系统判定轨迹异常
              ↓
        本次跑步无效，成绩取消
              ↓
        无申诉渠道，用户只能重跑
```

核心问题：正常跑步被频繁打断，弱网/漂移必然导致误判，用户完全不知道自己处于什么状态。

3.2 优化后（To-Be）

```
开始跑步 → GPS 正常采集轨迹
              ↓
        状态面板：🟢 绿色 - 正常跑步
              │
              ├─→ 检测到瞬时速度 >25km/h 或步频归零但继续移动
              │       ↓
              │   面板变红 🔴 弹出人脸识别
              │       ↓
              │   验证通过 → 继续跑步，面板恢复绿色
              │
              └─→ GPS 信号丢失
                      ↓
                  面板变黄 🟡 启动加速度计缓存补偿
                      ↓
                  3s 内恢复 → 自动补全轨迹，面板恢复绿色
                  3-20s 恢复 → 补全轨迹 + 标记低精度段
                  >20s 未恢复 → 面板变红 🔴 标记人工审核
                      ↓
                  跑步结束 → 成绩正常记录
```

核心改进：正常跑步零打扰，异常行为精准触发验证，弱网环境自动容错，状态全程可视。


4. 功能详细说明

F1：人脸识别策略优化（随机触发 → 异常触发）

做什么：将人脸识别弹窗从随机触发改为基于行为异常的智能触发，正常跑步全程不弹出。

触发条件：
| 触发场景 | 检测方式 | 说明 |
|---------|---------|------|
| 瞬时速度异常 | GPS 检测瞬时速度 >25km/h | 疑似骑车代跑 |
| 步频归零但持续移动 | 加速度计检测步频为 0 且位置持续移动 | 疑似将手机放在代跑设备上 |
| 正常跑步 | 不触发 | 全程无弹窗打扰 |

不触发场景：

- 配速 5-8 分钟/公里、步频稳定 → 正常跑步，不弹窗
- 短暂停顿（系鞋带、喝水）→ 步频归零但位置未移动，不触发
- 配速波动在合理范围内 → 不触发

交互原型：见 [prototype.html](prototype.html) · 方案一：人脸识别触发流程


F2：弱网定位补偿

做什么：GPS 信号短时丢失时，利用加速度计缓存数据自动补全轨迹，避免误判作弊。

补偿策略：
| 信号丢失时长 | 处理方式 | 用户体验 |
|-------------|---------|---------|
| 0-3 秒 | 自动补全轨迹 | 完全无感知 |
| 3-20 秒 | 线性插值补全 + 标记低精度段 | 面板变黄提示，成绩正常记录 |
| >20 秒 | 标记待人工审核 | 面板变红，成绩暂挂审核 |

技术原理：

- GPS 信号正常时持续采集轨迹坐标
- 信号丢失后启动加速度计记录步数和方向
- 信号恢复后读取缓存数据，线性插值补全轨迹坐标
- 低精度段标记供后续人工审核参考

交互原型：见 [prototype.html](prototype.html) · 方案二：弱网补偿流程


F3：三色状态提示面板

做什么：跑步过程中实时显示 GPS 信号状态和跑步基础数据，让用户清楚了解当前情况。

状态设计：
| 颜色 | 含义 | 触发条件 |
|------|------|---------|
| 🟢 绿色 | 正常跑步 | GPS 信号良好，轨迹正常采集 |
| 🟡 黄色 | 弱网补偿中 | GPS 信号弱，已启动缓存补偿 |
| 🔴 红色 | 需验证/异常 | 检测到异常行为需人脸验证，或信号丢失超过 20 秒 |

面板展示信息：

- 当前状态（颜色 + 文字说明）
- 跑步距离（公里）
- 当前配速（分钟/公里）
- 已用时间

交互原型：见 [prototype.html](prototype.html) · 方案三：状态面板展示


5. 验收标准

F1：人脸识别策略优化

| ID | 验收条件 | 优先级 |
|----|---------|--------|
| AC-01 | 正常跑步（配速 5-8 分钟/公里、步频稳定）全程不触发人脸识别 | P0 |
| AC-02 | 瞬时速度超过 25km/h 时触发人脸识别 | P0 |
| AC-03 | 步频归零且持续移动时触发人脸识别 | P0 |
| AC-04 | 短暂停顿（系鞋带、喝水、等红灯）不触发人脸识别 | P0 |
| AC-05 | 人脸识别弹窗显示明确的触发原因（如"检测到速度异常，请验证身份"） | P1 |

F2：弱网定位补偿

| ID | 验收条件 | 优先级 |
|----|---------|--------|
| AC-06 | GPS 信号丢失 3 秒内自动补全轨迹，用户无感知 | P0 |
| AC-07 | GPS 信号丢失 3-20 秒自动补全轨迹并标记低精度段 | P0 |
| AC-08 | GPS 信号丢失超过 20 秒标记为待人工审核 | P0 |
| AC-09 | 补偿期间面板显示黄色状态，提示"弱网补偿中" | P1 |
| AC-10 | 人工审核结果在 24 小时内反馈给用户 | P1 |

F3：三色状态提示面板

| ID | 验收条件 | 优先级 |
|----|---------|--------|
| AC-11 | 面板实时显示当前状态颜色（绿/黄/红） | P0 |
| AC-12 | 面板显示跑步距离、配速、用时三项基础数据 | P0 |
| AC-13 | 状态切换时面板颜色平滑过渡，延迟不超过 1 秒 | P1 |
| AC-14 | 红色状态提供明确的下一步操作指引（如"请进行人脸验证"） | P1 |

非功能验收标准

| ID | 验收条件 | 优先级 |
|----|---------|--------|
| AC-15 | 状态面板刷新延迟 ≤ 1 秒 | P1 |
| AC-16 | 加速度计缓存数据不占用超过 50MB 内存 | P1 |
| AC-17 | 支持 Android 和 iOS 双端 | P0 |


6. 风险、依赖与范围

6.1 风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| 异常触发阈值难以精确标定 | 中 | 部分骑车代跑可能未被识别 | 初期采用较保守的阈值，根据反馈迭代调整 |
| 加速度计数据在部分低端机型精度不足 | 中 | 弱网补偿效果打折扣 | 增加机型适配测试，低精度机型降低补偿时长上限 |
| 人脸识别改为异常触发后可能被学生利用 | 低 | 作弊率可能上升 | 保留随机抽检作为兜底（如 5% 概率随机触发） |
| 闪动校园官方不一定采纳方案 | 高 | 方案停留在纸面 | 本方案作为 PM 作品集项目，核心价值在于展示产品思维全流程 |

6.2 依赖

| 依赖 | 用途 | 备注 |
|------|------|------|
| 加速度计传感器 API | F2 步频检测和轨迹补偿 | Android/iOS 原生支持，无外部依赖 |
| GPS 定位精度 | F1 速度异常判定、F3 状态面板 | 受硬件和环境限制，需设定合理容差范围 |
| 闪动校园现有防作弊系统接口 | 方案集成 | 此为外部依赖，非本项目可控 |

6.3 范围

| 本次实现 | 本次不实现 |
|---------|-----------|
| 人脸识别触发策略优化（异常触发） | 人脸识别算法本身（使用现有方案） |
| 弱网 GPS 定位补偿（加速度计缓存） | 离线跑步模式（完全无网环境下跑步） |
| 三色状态提示面板 | 跑步社交功能（排行榜、分享等） |
| 人工审核兜底机制 | 申诉系统完整重构 |


7. 迭代计划

```
      第1周  第3周  第6周
V1     ████████████████
       (异常触发人脸识别 + 状态面板基础版)
V2                    ████████████████████
                      (弱网定位补偿 + 人工审核)
V3                                      ████████████
                                        (阈值调优 + 随机抽检兜底)
```

V1（前 4 周）：核心体验修复

- 人脸识别从随机触发改为异常触发（瞬时速度 + 步频检测）
- 三色状态面板基础版（绿色正常 + 红色异常）
- 依赖：加速度计和 GPS 基础 API
- 目标：正常跑步人脸识别触发次数从 3-5 次降至 0 次

V2（第 5-8 周）：弱网容错

- 加速度计缓存补偿机制上线
- 人工审核后台
- 状态面板增加黄色弱网补偿状态
- 依赖：加速度计缓存数据采集与插值算法
- 目标：弱网误判率从 62% 降至 15% 以下

V3（第 9-10 周）：精细化调优

- 基于实际跑步数据迭代阈值参数
- 增加 5% 随机抽检作为兜底策略
- 面板交互细节优化
- 目标：在保障风控效果的前提下持续降低误判率
