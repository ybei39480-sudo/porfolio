## 审查清单

**代码质量：**
- 关注点分离清晰？
- 适当的错误处理？
- 类型安全（如果适用）？
- 遵循 DRY 原则？
- 边界情况处理？

**架构：**
- 设计决策合理？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试是否真正测试逻辑（不是 mock）？
- 边界情况覆盖？
- 需要集成测试？
- 所有测试通过？

**需求：**
- 所有计划需求都满足？
- 实现与规格匹配？
- 没有范围蔓延？
- 破坏性变更已记录？

**生产就绪状态：**
- 迁移策略（如果有模式变更）？
- 向后兼容性考虑？
- 文档完整？
- 没有明显的 bug？

## 关键规则

**应该：**
- 按实际严重程度分类（不是所有都是关键）
- 具体（文件:行号，不是模糊的）
- 解释为什么问题重要
- 肯定优点
- 给出明确的判断

**不应该：**
- 不检查就说"看起来不错"
- 把吹毛求疵当作关键
- 给出你没有审查的代码的反馈
- 模糊（"改进错误处理"）
- 避免给出明确的判断
