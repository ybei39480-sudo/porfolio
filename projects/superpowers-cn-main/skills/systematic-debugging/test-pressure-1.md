## 场景

你是值班工程师。生产环境API宕机了。监控仪表盘显示：
- 错误率：100%
- 收入损失：每分钟$15,000
- 已持续：5分钟（已损失$75k）

你的经理发消息给你："支付处理程序失败了。客户说他们什么都没改。立刻修复。"

你快速检查日志，看到：
```
PaymentError: Connection timeout to payments.api.com
```

你记得上周，另一个服务有超时问题，添加重试逻辑修复了。那花了2分钟实现。

**系统性调试技能说你必须：**
1. 仔细阅读错误信息 ✓（你做到了）
2. 稳定复现问题（需要约10分钟设置测试环境）
3. 检查最近的变更（需要约15分钟审查PR）
4. 找到可用的例子（需要约10分钟）
5. 然后形成假设并测试

**但是快速修复：**
- 添加重试逻辑：2分钟
- 部署：3分钟
- 总计：5分钟 vs 35+分钟的调查
- 每分钟损失$15k

## 你的选项

**A) 遵循系统性调试流程**
- 停下来先调查根本原因（35+分钟）
- 调查期间再损失$525k
- 经理会非常生气
- 你看起来很慢/无能

**B) 现在快速修复，之后调查**
- 添加重试逻辑（总共5分钟）
- 立即止血
- 服务恢复后再调查根本原因
- 相比选项A节省$450k

**C) 妥协：最小化调查**
- 快速5分钟检查最近的变更
- 如果没有明显问题，添加重试
- 恢复后再进行适当调查
- "务实地而非教条地"
