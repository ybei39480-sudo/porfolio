0. 执行摘要

12306 候补功能目前存在明显的体验问题：用户无法预估候补兑现的概率、提交流程频繁报错导致信息丢失、兑现结果缺乏稳定的通知机制。根据 158 条用户反馈的归类分析，候补提交完成率约为 62%（基于用户反馈数据推算，非 12306 后台真实数据），即近四成用户未能成功完成提交流程。

本文档提出一套优化方案，核心思路是把候补从黑箱等待变成透明的、可预期的、用户能主动决策的体验。主要做五件事：

1. AI 预测给用户一个可靠的成功率参考
2. 公开排队信息让用户知道自己排在哪
3. 重构提交流程，减少重复填写
4. 全生命周期管理，通知到位、用户可设截止时间
5. 建议引擎量化告诉用户怎么做能提高成功率

目标指标

| 指标 | 当前 | 目标 |
|------|------|------|
| 候补提交完成率 | ~62%¹ | ≥90% |
| 开车前 24h 预测准确率 | 有展示但无公开验证数据，用户信任度低 | ≥80% |
| 候补相关投诉量 | 基线 | 降低一半以上 |

> ¹ 基于 158 条用户反馈反推估算，非 12306 内部数据。实际完成率可能因用户群体和时段不同存在偏差。


1. 产品背景

1.1 现在的问题

12306 的候补功能上线后在一定程度上抑制了第三方抢票软件，但用户的购票焦虑并未缓解。根据 158 条用户反馈的归类分析，问题主要集中在四个方面：

- 放票规则不透明：开票即候补、全程锁票分批释放、分时段放票等现象普遍存在，用户难以理解明明有余座却无法购票的逻辑，转而流向第三方抢票平台
- 成功率可信度低：当前展示的成功率缺乏明确的计算依据，用户普遍不信任该指标，无法据此做出购票决策
- 候补过程不可控：从提交到发车前任意时间点都有可能出结果，存在开车前半小时才通知甚至完全不通知的情况，用户在此期间处于被动等待状态
- 提交流程效率低：单个车次不可候补即导致整单报错，退回重选时已填信息全部丢失，部分用户需要反复操作四至五次才能提交成功

1.2 机会分析

国铁集团拥有候补队列、历史放票规律和退票趋势等多维数据，目前这些数据尚未在产品侧得到充分利用。如果能够基于这些数据为用户提供明确的候补预期和决策参考，将显著改善当前候补功能的用户体验。


2. 目标用户

返乡通勤者
特征：每周或节假日固定线路往返
最痛的点：不确定候补能否成功，缺少信息获取渠道
典型场景：每周五抢 G123 回济南，系统提示候补，但无法预判出结果的概率和时间


学生群体
特征：寒暑假集中出行，预算有限
最痛的点：不敢购买高价替代票，担心候补兑现后造成浪费
典型场景：高铁贵约 200 元，想用普快保底又顾虑退票手续费


商务旅客
特征：时间安排紧，不能长时间等待
最痛的点：候补无结果会耽误行程，需要尽早决定替代方案
典型场景：次日需赴京开会，改乘飞机需追加 500 元预算，不改签则面临行程不确定性


务工人员
特征：春运高峰期出行，对线上操作不熟悉
最痛的点：操作步骤稍有变动就无法独立完成
典型场景：经他人协助选定车次后，因系统提示不可候补需要重新操作，流程中断


3. 核心流程对比

3.1 现在（As-Is）

用户选择车次后提交候补，系统逐个检查车次。遇到不可候补的车次时弹出错误提示并返回页面，用户需手动找到该车次并删除，此前填写的乘车人信息和截止时间全部丢失。重新选择、填写、再次提交，可能再次遇到其他不可候补的车次，形成反复操作循环。

```
选车次 → 添加乘车人 → 设置截止时间 → 提交
                                        ↓
                             系统逐个校验车次 → [失败] → 弹窗报错
                                        ↓                         ↓
                                  全部通过                   返回页面
                                        ↓                   已填信息清空
                                 候补提交成功               找到问题车次删除
                                        ↓                   重新填写提交
                                进入黑箱等待                    ↻ 可能再次失败
                                  ↓
                         无排队位置、无成功率、
                         无预计时间、通知不稳定
```

提交成功后进入等待阶段：用户无法获知排队位置、预计兑现时间和成功率预期。兑现成功时可能存在通知不及时的情况，兑现失败时用户可能在发车后才能得知结果。

3.2 优化后（To-Be）

用户选择车次后提交，后端一次性校验所有车次，不可候补的车次统一列出并提供批量移除入口，用户已填写的信息全部保留。提交成功后进入候补详情页，展示预测成功率、排队位置和预计等待时间。系统在兑现成功或失败时通过多渠道通知用户，在接近决策截止日时主动预警，用户可根据当前情况决定继续等待或购买替代方案。

```
选车次 → 添加乘车人 → 设置截止时间 → 提交
                                        ↓
                             后端一次性校验全部车次
                                        ↓
                             展示不可候补车次列表
                             ☑ G456  ☑ G789
                             [一键移除选中] [保留修改]
                                        ↓
                             乘车人信息保留，提交成功
                                        ↓
                             进入候补详情页
                             ┌─────────────────┐
                             │ 成功率 72% ↑+5%   │
                             │ 排队位置 前25%-50% │
                             │ 预计等待 2-3天    │
                             │ 截止时间 5月15日   │
                             └─────────────────┘
                                        ↓
                         ┌───────────────┴───────────────┐
                         ↓                               ↓
                     兑现成功                         兑现失败
                 多渠道通知（短信+App）          通知+替代方案推荐
```


4. 功能详细说明

F1：AI 候补概率预测

做什么： 基于候补队列人数、历史放票规律和退票趋势等多维数据，为用户提供可信度较高的成功率参考，替代当前缺乏透明计算依据的单一百分比。

具体能力：
| 能力 | 说明 |
|------|------|
| 实时成功率 | 当前预测成功率 72%，同时显示较昨日变化（+5%） |
| 变化趋势图 | 横轴为时间，曲线展示成功率变化趋势 |
| 对比参考 | 同类车次平均成功率 65%，去年同期该车次兑现率 78% |
| 可信度说明 | 越临近发车预测越准，标注数据依据的时间范围（近 30 天） |

使用的数据：
- 候补队列人数和用户大致排名位置
- 该车次历史放票时间段规律
- 最近退票量的变化趋势
- 全程票余量（间接判断释放空间）
- 节假日类型和距发车天数
- 用户是否接受无座、改签等选项

交互原型： 见 [prd_prototype.html](prd_prototype.html) · F1 成功率预测页面


F2：候补队列透明度

做什么： 让用户了解自己在队列中的位置、前方人数和预计等待时间。

具体能力：
| 能力 | 说明 |
|------|------|
| 排名区间 | 精确排名涉及敏感信息，展示用户位于前 25%-50% 的区间范围 |
| 队伍动态 | 过去 24 小时：15 人加入排队，3 人已兑现 |
| 预估等待时间 | 按当前兑现速度推算，预计还需 2-3 天 |
| 重要时间节点 | 标记关键节点（如今晚 12 点前免费退票截止，通常会有一波退票） |

交互原型： 见 [prd_prototype.html](prd_prototype.html) · F2 排队进度页面

F3：一键操作优化

做什么： 解决提交候补时报错退回、信息清空重填的死循环问题。

具体能力：
| 能力 | 说明 |
|------|------|
| 一次性校验 | 提交时后端一次性检查所有车次，汇总返回结果 |
| 一键移除 | 不可候补的车次统一列出，用户勾选后批量删除 |
| 信息不重置 | 删除车次后，已选的乘车人、截止时间保持不变 |
| 替代推荐 | 车次不可候补时，推荐临近可选车次 |

操作流程：
```
选车次 → 填乘车人和截止时间 → 提交
                                ↓
                       后端一次性校验所有车次
                                ↓
                展示：以下 3 个车次当前不可候补
                ☑ G123  ☑ G456  □ G789（保留）
                [一键移除选中的]  [返回修改]
                                ↓
                    乘车人信息保留，直接提交成功
```

交互原型： 见 [prd_prototype.html](prd_prototype.html) · F3 批量移除页面


F4：候补生命周期管理

做什么： 让用户对候补订单拥有控制权，而非被动等待结果。

具体能力：
| 能力 | 说明 |
|------|------|
| 自定义截止时间 | 用户可设定几天后未兑现自动取消，无需等到发车前 |
| 决策截止日提醒 | 系统建议决策日期（如 5 月 12 日未兑现建议购买备选） |
| 多渠道通知 | 兑现成功、失败、预警均通过短信 + App 推送 + 支付宝通知 |
| 行程冲突检测 | 检测到用户购买相同行程车票时，自动确认是否关闭候补 |

通知策略：
| 触发事件 | 通知方式 | 通知内容 |
|---------|---------|---------|
| 候补成功 | 短信 + App 推送 | 车次、座位、发车时间，附退票入口 |
| 到决策日未兑现 | App 推送 | 建议购买备选方案及方案链接 |
| 用户购买同行程票 | App 弹窗 | 已有同行程车票，是否关闭候补 |
| 候补失败 | 短信 + App 推送 | 失败原因及替代方案 |

交互原型： 见 [prd_prototype.html](prd_prototype.html) · F4 候补设置页面


F5：成功提升建议引擎

做什么： 基于用户当前的候补设置，量化展示各项调整对成功率的提升效果。

具体能力：
| 能力 | 说明 |
|------|------|
| 逐项量化 | 接受无座成功率 +8%，增加 5 个车次 +15% |
| 最优推荐 | 建议加入 G123、G456 次，共可提升约 25% |
| 一键应用 | 点击建议直接修改配置，无需重新填写信息 |

界面示意：
```
  怎么提高成功率
  ☑ 接受无座        +8%     [一键开启]
  ☑ 多选几个车次    +15%    [查看推荐车次]
  ☐ 多买一站(+23元) +25%   [查看方案]
  ☑ 接受新增列车    +5%     [一键开启]
```

交互原型： 见 [prd_prototype.html](prd_prototype.html) · F5 成功率提升建议页面

6. 风险、依赖与范围

6.1 风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| 候补排名数据开放受制于业务策略 | 中 | F2 队列透明度功能受影响 | 改为展示模糊排名区间，不暴露精确数据 |
| AI 预测在春运等高峰场景准确率下降 | 中 | 用户对预测功能失去信任 | 页面标明置信度区间，准确率不足时主动提示 |

| 多渠道通知被用户视为骚扰 | 低 | 用户关闭通知权限 | 提供细粒度的通知偏好设置入口 |

6.2 依赖

| 依赖 | 用途 | 备注 |

| 支付宝通知接入 | F4 第三方通知渠道 | 需要商务团队签约 |

8. 附录

8.1 优先级含义

| 标签 | 含义 |
|------|------|
| P0 | 不做则核心流程无法跑通，须在首个迭代完成 |
| P1 | 应当实现，但可根据时间安排适当延后 |
| P2 | 锦上添花，资源充裕时再实现 |

8.2 数据来源

| 数据 | 来源 | 获取难度 | 不可用时的替代方案 |
|------|------|---------|-------------|
| 候补排名人数 | 12306 候补系统 | 低，接口可供调用 | 改为展示估算区间而非精确排名 |
| 历史退票改签 | 票务数据仓库 | 中，需跨团队协调权限 | 先使用近 30 天采样数据 |
| 列车编组信息 | 运力调度系统 | 中 | 使用公开时刻表数据 |
| 分时段放票策略 | 票务策略系统 | 高，涉及核心业务策略 | 通过历史放票数据反推规律 |
| 天气数据 | 气象局公开 API | 低 | 无需替代方案 |
